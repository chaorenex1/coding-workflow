# Coding Workflow 仓库知识总览

## 1. 背景与范围

本文档提炼的是仓库级知识，不讨论具体函数实现。目标是帮助读者理解该项目如何通过技能、命令与钩子组织 AI 开发流程。

## 2. 关键事实

1. 这是一个面向 Claude Code 的插件型工作流仓库，定位是从需求到实现的自动化开发辅助系统。
2. 项目元信息集中在 `.claude-plugin/plugin.json`，当前版本为 `3.1.0`，并声明 MIT 许可证。
3. `hooks/hooks.json` 定义了会话启动与工具调用前钩子，其中会话启动会执行依赖检查脚本。
4. `README.md` 明确给出了能力边界：由 Skills、Agents、Commands 共同构成系统能力。
5. 仓库结构采用“能力分层目录”方式组织，`skills/`、`agents/`、`commands/` 是核心入口。

## 3. 稳定模式

1. **声明式元信息模式**
   - 项目与组件都倾向于通过 Markdown 或 JSON 声明元信息，而不是把规则散落在脚本中。
   - 好处是易读、可审计、便于自动发现。

2. **目录即能力边界模式**
   - 技能、代理、命令分目录管理，形成“职责天然分离”。
   - 新能力通常通过新增目录与说明文件接入，而不是改动中心化注册代码。

3. **前置防护模式**
   - 通过 PreToolUse 钩子在动作前执行校验逻辑，体现“先约束后执行”的工程策略。
   - 这类机制适合统一策略控制、风险拦截与行为审计。

4. **文档先行模式**
   - README 承担对外能力地图，技能文档承担对内执行规范。
   - 文档在该仓库中不仅是说明，更是“行为协议”。

## 4. 决策链条

1. **背景**
   - 目标是让 AI 参与开发时具备可复用、可组合、可扩展的工作流能力。

2. **约束**
   - 能力来源多样（技能、代理、命令、脚本），如果缺乏统一组织将导致维护复杂度上升。
   - 需要兼顾可读性、可扩展性和自动化执行。

3. **可选方案**
   - 方案 A：集中式单体脚本，把所有流程写入一个大入口。
   - 方案 B：模块化目录结构，以文档和配置驱动自动发现与执行。

4. **最终选择**
   - 采用方案 B：将能力拆分到 `skills/`、`agents/`、`commands/`，配合 hooks 形成运行时约束。

5. **结果与影响**
   - 正向影响：扩展新能力成本较低，协作边界更清晰。
   - 代价：文档与结构一致性要求更高，需要持续维护规范。

## 5. 实践建议

1. 新增能力优先遵循现有目录边界，不要跨目录混合职责。
2. 当新增自动化动作时，优先评估是否应纳入 hooks 体系。
3. 任何“跨组件协作逻辑”都应有对应的知识文档，避免仅存在于口头约定。
